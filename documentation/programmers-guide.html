The OPSL Programmer's Guide

Everything you need to know to write effective OPSL scripts and libraries — from the four basic instructions to advanced patterns, debugging, and best practices.

What is OPSL?

OPSL (One Programming/Scripting Language) is a custom scripting language designed to orchestrate code execution across multiple programming languages. Instead of writing everything in one language, you write a simple script that tells different libraries — written in JavaScript, Python, C, or even other OPSL files — what to do and when.

Think of OPSL as a conductor of an orchestra. The conductor doesn't play any instrument. Instead, they point to the violinist when it's time for strings, cue the drummer for percussion, and signal the flutist for melody. OPSL does exactly that — it points to libraries and tells them when to execute.

Key insight: OPSL itself has no loops, no conditionals, and no variables of its own. It's purely sequential — one instruction after another. Loops and conditionals are achieved through library logic and the CALL opsl.functionCall mechanism.

The 4 Instructions

OPSL has exactly four instructions. That's it. Every OPSL script is just a sequence of these four commands:

InstructionFormatWhat it does
IMPORTIMPORT [lang] nameLoads a library into the program
VARIABLEVARIABLE lib.var = "value"Stores a value in a library
ASSIGNASSIGN lib.target = source.varCopies a value between libraries
CALLCALL lib.functionNameExecutes a function in a library

These four instructions are the entire language. There are no keywords beyond these. There are no operators, no expressions, no control flow. Just these four commands, executed one after another from top to bottom.

IMPORT — Deep Dive

What it does

IMPORT [lang] name loads a library and makes it available for subsequent VARIABLE, ASSIGN, and CALL instructions.

Syntax breakdown

IMPORT [js] greeting ↑ ↑ lang name IMPORT [opsl] helper ↑ ↑ lang name

How OPSL finds the library file

When you write IMPORT [js] greeting, OPSL searches for the file in two locations, in this order:

  1. Project root: projectFolder/greeting.js
  2. Local folder: projectFolder/opsl-local/greeting/greeting.js

If the language is [opsl], it looks for .opsl files instead. The first match wins.

What happens when IMPORT runs

IMPORT does three things behind the scenes:

  1. Locates the file on disk using the search rules above
  2. Starts the library — for JS, spawns node library.js as a separate process
  3. Waits for ready — the library must print "loaded and registered" to stdout

Once all three steps complete, the library is ready to receive VARIABLE, ASSIGN, and CALL instructions.

Important rules

  • Each library is imported only once. A second IMPORT of the same name is silently ignored.
  • Libraries must be imported before use. You can't call a function in a library that hasn't been imported yet.
  • Import order matters. Libraries are started sequentially, so import order affects initialization timing.

VARIABLE — Deep Dive

What it does

VARIABLE lib.var = "value" stores a value in a library's namespace on the TCP server.

Syntax breakdown

VARIABLE test.myVar = "hello world" ↑ ↑ ↑ lib var value VARIABLE add.counter = "0" ↑ ↑ ↑ lib var value

What actually happens

When OPSL encounters this instruction:

  1. It looks up the library test in its registry of imported libraries
  2. It sends a TCP message to the server: id:test:set:myVar:hello world
  3. The server stores: clients["test"].values["myVar"] = "hello world"
  4. The server responds: id:ok

The value is now stored on the TCP server, associated with the test library.

Important rules

  • Values are always strings. Even if you write "42", it stays as the string "42", not the number 42.
  • Quotes are required. The value must be enclosed in double quotes: = "value"
  • Each VARIABLE overwrites. Setting the same variable twice replaces the previous value.
  • The library itself may not know about it yet. The variable is stored on the server. The library must call TCPClient.get() to read it.
Common confusion: VARIABLE stores a value on the TCP server. It does NOT create a local variable inside the JavaScript library. The library must explicitly read it via TCPClient.get('myVar').

ASSIGN — Deep Dive

What it does

ASSIGN target.name = source.var copies a value from one library's namespace to another's.

Syntax breakdown

ASSIGN test.result = add.output ↑ ↑ ↑ ↑ target target source source lib var lib var ASSIGN opsl.consoleLog = ifJS.number ↑ ↑ ↑ ↑ target target source source lib var lib var

What actually happens (two TCP round-trips)

  1. GET from source: OPSL sends id:add:get:output → server returns id:42
  2. SET to target: OPSL sends id:test:set:result:42 → server stores it

The value "42" has now been copied from add.output to test.result.

Important rules

  • Both libraries must be imported. If either library is missing, ASSIGN silently does nothing.
  • OPSL libraries are skipped. OPSL libraries don't store variables on the TCP server, so you can't ASSIGN from or to them.
  • The source variable must exist. If add.output was never set, you get an empty string "".
  • This is a copy, not a reference. Changing the source after ASSIGN doesn't affect the target.

CALL — Deep Dive

What it does

CALL lib.functionName executes a function. What happens next depends on the library's language.

For JavaScript libraries

The function call travels through TCP:

  1. OPSL sends id:lib:call:lib:functionName to the server
  2. Server looks up the library's local server port
  3. Server connects to the library's local server and sends call:functionName
  4. The library executes the function
  5. The result travels back through the server to OPSL

Functions have no parameters and return nothing. They simply execute. If they need data, they read from the TCP server via TCPClient.get(). If they produce data, they write to the TCP server via TCPClient.set().

For OPSL libraries

CALL executes a nested .opsl file:

  1. OPSL looks for opsl-local/lib/functionName.opsl
  2. If found, it creates a new Runtime instance
  3. The new runtime shares all imported libraries from the parent
  4. It processes the nested .opsl file line by line
  5. When done, control returns to the parent script

Special calls

Two OPSL library calls have special behavior:

CALL opsl.functionCall → Reads the "functionVar" variable from the opsl library → Parses it as "library.functionName" → Executes that function dynamically CALL opsl.consoleLog → Reads the "consoleLog" variable from the opsl library → Prints it to the console

How Data Flows Between Libraries

Understanding data flow is the key to writing effective OPSL. Here's the complete picture:

The TCP server is the shared memory. Libraries don't talk to each other directly. They all talk to the TCP server, which acts as a central message hub and data store.

The flow of a VARIABLE instruction

OPSL script TCP Server JS Library | | | | VARIABLE test.x = "hi" | | |------------------------->| | | id:1:test:set:x:hi | | | | stores: | | | test.values.x = "hi" | |<-------------------------| | | id:1:ok | | | | |

The flow of a CALL instruction

OPSL script TCP Server JS Library (port 4001) | | | | CALL test.sayHello | | |------------------------->| | | id:2:test:call:test:sayHello | | | connects to port 4001 | | |---------------------->| | | call:sayHello | | | | executes sayHello() | |<----------------------| | | result | |<-------------------------| | | id:2:result | |

The flow of an ASSIGN instruction

OPSL script TCP Server | | | ASSIGN test.y = add.x | |------------------------->| | id:3:add:get:x | |<-------------------------| | id:3:"42" | |------------------------->| | id:4:test:set:y:42 | |<-------------------------| | id:4:ok |

Writing JavaScript Libraries

Every JS library follows this template. This is the only pattern:

// 1. Require the TCP client var TCPClient = require('opsl-tcp-client'); // 2. Define functions — no parameters allowed function myFunction() { // Read state from TCP server var x = await TCPClient.get('myVar'); // Do work console.log("Value is: " + x); // Write state back to TCP server await TCPClient.set('result', 'done'); } // 3. Initialize — no port needed (OS assigns automatically) (async () => { await TCPClient.init('libraryName'); await TCPClient.set('initialVar', 'value'); await TCPClient.registerFunction('myFunction', myFunction); console.log("loaded and registered"); })();

Every line explained

Line 1: require

var TCPClient = require('opsl-tcp-client'); — This is a globally installed npm package. No path needed. It gives you the 5 methods: init(), set(), get(), registerFunction(), and callRemote().

Lines 4-12: Functions

Functions have no parameters and no returns. If a function needs data, it calls TCPClient.get(). If it produces data, it calls TCPClient.set(). The function body can be async — use await for TCP operations.

Lines 15-19: Initialization

This is an async IIFE (Immediately Invoked Function Expression). It runs when the file is loaded. The four steps are:

  1. TCPClient.init('name') — connects to server, starts local server (port auto-assigned)
  2. TCPClient.set() — set initial variables (optional)
  3. TCPClient.registerFunction() — make functions callable from OPSL
  4. console.log("loaded and registered") — signals to OPSL that the library is ready

Function Rules

Functions in OPSL libraries follow strict rules:

RuleExplanation
No parametersFunctions are always called with zero arguments. function greet() — not function greet(name)
No returnsReturn values are ignored. Use TCPClient.set() to communicate results.
Can be asyncUse async function and await TCPClient.get/set
Must be registeredOnly functions passed to registerFunction() are callable from OPSL
Can call other functionsFunctions can call each other internally — just not from OPSL with args

Why no parameters?

Because OPSL is designed around shared state via the TCP server. Instead of passing arguments, the caller sets variables on the server, the function reads them, does work, and writes results back. This makes the data flow explicit and traceable.

Variable Rules

The two types of variables

There are actually two places variables can live:

  1. On the TCP server — set by OPSL's VARIABLE instruction or by TCPClient.set() inside a library
  2. Inside a library's local memory — local JS variables like var counter = 0
Critical: Local variables are not visible to other libraries or to OPSL. Only TCP server variables are shared. If you modify a local variable inside a function, other libraries will never see the change.

When to use TCPClient.get() inside functions

Always use TCPClient.get() at the start of a function if you need the current value of a variable. The value on the server may have been changed by an ASSIGN instruction from OPSL since the function last ran.

// CORRECT: Always read from server async function add() { var number = await TCPClient.get('number'); // Latest value number = String(Number(number) + 1); await TCPClient.set('number', number); // Update server } // WRONG: Local variables become stale var number = "0"; // Only initialized once when library loads async function add() { number = String(Number(number) + 1); // Uses stale local value! await TCPClient.set('number', number); }

Writing OPSL Libraries

OPSL libraries are simply .opsl files that get executed as nested scripts. They follow the same 4-instruction language.

How they work

When OPSL executes CALL myHelper.doSomething and myHelper is an OPSL library:

  1. OPSL looks for opsl-local/myHelper/doSomething.opsl
  2. Creates a new Runtime instance (nested)
  3. The nested runtime shares all imported libraries from the parent
  4. Processes the .opsl file line by line
  5. When done, returns control to the parent

Why use OPSL libraries?

  • Reusability: Extract common OPSL instruction sequences into named functions
  • Organization: Break large scripts into smaller, manageable pieces
  • Dynamic dispatch: The CALL opsl.functionCall mechanism lets you choose which OPSL function to execute at runtime

Common Patterns

Pattern 1: Set up, then execute

IMPORT [js] processor VARIABLE processor.mode = "fast" VARIABLE processor.input = "data.txt" CALL processor.run

Set configuration variables first, then call the function that reads them.

Pattern 2: Chain libraries

IMPORT [js] reader IMPORT [js] transformer IMPORT [js] writer CALL reader.read ASSIGN transformer.input = reader.output CALL transformer.transform ASSIGN writer.input = transformer.output CALL writer.write

Data flows: reader → transformer → writer, using ASSIGN as the pipe.

Pattern 3: Conditional branching via OPSL

IMPORT [js] checker IMPORT [opsl] opsLib VARIABLE checker.value = "test" CALL checker.check ASSIGN opsLib.branch = checker.result CALL opsLib.routeBranch

The JS library decides which branch to take, stores it, and the OPSL library reads it and routes accordingly.

Loops

OPSL has no for or while keywords. Loops are achieved through dynamic dispatch:

How the loop pattern works

IMPORT [js] counter IMPORT [js] condition IMPORT [opsl] loopBody VARIABLE condition.trueReturn = "" VARIABLE condition.falseReturn = "loopBody.repeat" ASSIGN condition.input = counter.value CALL condition.check ASSIGN opsl.functionVar = condition.output CALL opsl.functionCall

The pattern:

  1. A JS condition library checks if the loop should continue
  2. If yes: it sets output = "loopBody.repeat"
  3. If no: it sets output = "" (empty = stop)
  4. CALL opsl.functionCall reads functionVar and executes the named function
  5. If it was "loopBody.repeat", the nested OPSL file runs and calls opsl.functionCall again
  6. The loop repeats until condition returns empty string
See it in action: The LoopTest2 project demonstrates a complete counting loop that counts from 0 to 1000 using this pattern.

Conditionals

Conditionals are also achieved through library logic:

IMPORT [js] validator IMPORT [opsl] handlers VARIABLE validator.input = testData CALL validator.validate ASSIGN handlers.branch = validator.output CALL handlers.routeBranch

The JS library evaluates the condition and stores which branch to take. The OPSL handler library reads that branch and executes the appropriate nested script.

Debugging

Console output

Every library's console.log() is prefixed with the library name:

[test] test.js loaded and registered [add] add.js loaded and registered [ifJS] 0 [ifJS] 1 [ifJS] 2

Common issues and how to fix them

SymptomCauseFix

Enabling verbose output

The runtime prints each instruction as it executes. To see more detail, uncomment the debug console.log() statements in runtimeTCP/instruction-handler.js.

Best Practices

1. Import all libraries at the top

Don't scatter IMPORT instructions throughout your script. Import everything first, then use them.

// GOOD IMPORT [js] reader IMPORT [js] processor IMPORT [js] writer VARIABLE reader.file = "data.txt" CALL reader.read ...

2. Name variables descriptively

Use clear variable names that describe their purpose, not their type.

// GOOD VARIABLE processor.inputFile = "data.txt" VARIABLE processor.outputMode = "json" // BAD VARIABLE processor.x = "data.txt" VARIABLE processor.y = "json"

3. Set variables before calling functions

Functions read from the TCP server. Make sure the data is there before calling.

// GOOD VARIABLE processor.input = "hello" CALL processor.process // BAD — function may read stale data CALL processor.process VARIABLE processor.input = "hello"

4. Use ASSIGN to chain operations

When one library produces output that another consumes, use ASSIGN explicitly.

CALL producer.generate ASSIGN consumer.input = producer.output CALL consumer.consume

5. Keep libraries focused

Each library should do one thing well. A library named processor should only process — not read files, not write output, not validate input.

Performance Tips

1. Minimize TCP round-trips

Each VARIABLE, ASSIGN, and CALL costs one or more TCP round-trips (~0.01-0.17ms). Reduce the number of instructions in tight loops.

2. Batch variable sets in libraries

If a function needs to set multiple variables, do it in one function call rather than multiple VARIABLE instructions from OPSL.

3. Prefer JS libraries over OPSL libraries for loops

OPS libraries spawn nested runtimes, which adds file I/O overhead. For tight loops, do the iteration in JS and call OPSL only when needed.

Common Mistakes

Mistake 1: Forgetting to register functions

// WRONG function myFunc() { console.log("hi"); } // Never registered — CALL will fail // RIGHT function myFunc() { console.log("hi"); } await TCPClient.registerFunction('myFunc', myFunc);

Mistake 2: Using local variables instead of TCP variables

// WRONG: counter only initialized once var counter = 0; async function increment() { counter++; // Local — nobody else sees this } // RIGHT: use TCP server async function increment() { var val = await TCPClient.get('counter'); val = String(Number(val) + 1); await TCPClient.set('counter', val); }

Mistake 3: Calling functions before importing

// WRONG CALL greeting.sayHello // greeting not imported yet IMPORT [js] greeting // RIGHT IMPORT [js] greeting CALL greeting.sayHello

Mistake 4: Expecting ASSIGN to work with OPSL libraries

// This silently does nothing — OPSL libs don't store on TCP server ASSIGN opsHelper.result = add.output

Full Working Example

Here's a complete OPSL project that reads input, transforms it, and outputs the result:

project/test.opsl

IMPORT [js] reader IMPORT [js] transformer IMPORT [js] writer VARIABLE reader.fileName = "input.txt" CALL reader.read ASSIGN transformer.input = reader.content VARIABLE transformer.mode = "uppercase" CALL transformer.transform ASSIGN writer.content = transformer.output CALL writer.write

project/opsl-local/reader/reader.js

var TCPClient = require('opsl-tcp-client'); async function read() { var fileName = await TCPClient.get('fileName'); console.log('Reading: ' + fileName); // Simulated — in real code, use fs.readFileSync await TCPClient.set('content', 'hello world'); } (async () => { await TCPClient.init('reader'); await TCPClient.registerFunction('read', read); console.log("loaded and registered"); })();

project/opsl-local/transformer/transformer.js

var TCPClient = require('opsl-tcp-client'); async function transform() { var input = await TCPClient.get('input'); var mode = await TCPClient.get('mode'); if (mode === 'uppercase') { input = input.toUpperCase(); } await TCPClient.set('output', input); } (async () => { await TCPClient.init('transformer'); await TCPClient.registerFunction('transform', transform); console.log("loaded and registered"); })();

project/opsl-local/writer/writer.js

var TCPClient = require('opsl-tcp-client'); async function write() { var content = await TCPClient.get('content'); console.log('Output: ' + content); } (async () => { await TCPClient.init('writer'); await TCPClient.registerFunction('write', write); console.log("loaded and registered"); })();
