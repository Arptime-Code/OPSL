TCP Communication System

How libraries communicate through the central TCP server — from connection to message routing to function calls.

Overview

Every OPSL library runs as its own Node.js process. The only way they communicate is through TCP. A central server on port 3000 routes all messages between clients.

┌──────────┐ TCP ┌──────────────┐ TCP ┌──────────┐ │ Library │◄─────────►│ TCP Server │◄─────────►│ Library │ │ test │ port 3000│ (port 3000) │ port 3000│ add │ │ port:4001│ │ │ │ port:4010│ └──────────┘ └──────────────┘ └──────────┘ │ │ │ │ Local Server │ Local Server │ │ (port 4001) │ (port 4010) │ └───────────────────────┼────────────────────────┘ │ Function calls flow through the server → target's local server

Each library has two TCP connections:

  • Client connection to the main server (port 3000) — for set/get/register/callRemote
  • Local server on a unique port (4001+) — for receiving function calls from other libraries

Server Architecture

The TCP server lives in tcp-server-v3/server/ — 6 files working together:

FileRole
index.jsCreates net.createServer on port 3000. One parser per socket.
handler.jsDispatches messages by action: init/set/get/reg/call
parser.jsBuffers incoming TCP data, splits on newlines, parses format
client-registry.jsIn-memory store: {name: {values:{}, functions:[], port:N}}
protocol.jsFormats responses: respond(socket, requestId, data)
remote-call.jsRoutes function calls between clients via their local servers

Server Message Handling

// When a message arrives: // "5:test:set:myVar:hello" // 1. parser.js buffers and splits: buffer += chunk; lines = buffer.split('\n'); message = "5:test:set:myVar:hello"; // 2. parser.js parses: {requestId: "5", name: "test", action: "set", key: "myVar", extra: "hello"} // 3. handler.js dispatches: if (action === 'set') { clients[name].values[key] = extra; // clients.test.values.myVar = "hello" respond(socket, "5", "ok"); // writes "5:ok\n" back }

Client Architecture

The client library lives in tcp-server-v3/client/ — 6 files:

FileRole
api.jsPublic API: init(), set(), get(), registerFunction(), callRemote()
connection.jsPersistent TCP connection to server: connectToServer(), sendRaw()
local-server.jsLocal TCP server for receiving function calls
protocol.jsBuilds request messages with incrementing IDs
responses.jsParses incoming responses, resolves matching promises
index.jsRe-exports api.js

Persistent Socket (The Big Optimization)

The runtime (tcp-workers.js) maintains one persistent TCP connection to the server. This is the single biggest performance improvement in the entire system.

How It Works

// In tcp-workers.js: var runtimeSocket = null; // Starts as null var pendingRequests = {}; // Maps request IDs to resolve functions var nextRequestId = 0; // Increments for each request function getRuntimeSocket() { if (runtimeSocket) return Promise.resolve(runtimeSocket); // REUSE return new Promise(function(resolve) { runtimeSocket = net.connect(3000, function() { resolve(runtimeSocket); }); runtimeSocket.on('data', handleIncomingData); // Single handler }); }

Key insight: The socket is created once (lazy initialization), then reused forever. No reconnection overhead per instruction.

Request-Response Matching

// Sending a request: function sendRuntimeRequest(name, action, key, value) { return new Promise(function(resolve) { getRuntimeSocket().then(function(socket) { var id = String(++nextRequestId); // Unique ID pendingRequests[id] = resolve; // Store resolve function var msg = id + ':' + name + ':' + action + ':' + key; if (value !== undefined) msg += ':' + value; socket.write(msg + '\n'); // Send and wait }); }); } // Receiving the response: function handleIncomingData(chunk) { socketBuffer += chunk.toString(); var lines = socketBuffer.split('\n'); socketBuffer = lines.pop() || ''; for (var i = 0; i < lines.length; i++) { var colonIdx = lines[i].indexOf(':'); var reqId = lines[i].substring(0, colonIdx); var data = lines[i].substring(colonIdx + 1); if (pendingRequests[reqId]) { pendingRequests[reqId](data); // Unblock the await! delete pendingRequests[reqId]; // Prevent memory leaks } } }

Message Protocol

All messages are newline-delimited strings with colon-separated fields:

DirectionFormatExample
Requestid:name:action:key[:value]\n5:test:set:myVar:hello\n
Responseid:data\n5:ok\n

The 5 Actions

ActionPurposeServer Behavior
initRegister a clientStores client's port number
setStore a variableclients[name].values[key] = value
getRetrieve a variableResponds with stored value or empty string
regRegister a functionAdds name to clients[name].functions[]
callCall a remote functionRoutes to target's local server

Performance Details

ScenarioTime
Persistent socket instruction~0.06ms
If we reconnected per instruction~100ms (1600× slower)
TCP round-trip (localhost)~0.02ms
Server message dispatch~0.01ms
Buffer parsing~0.005ms
Bottom line: The persistent socket saves ~99.94ms per instruction. Over 1000 iterations, that's 99.9 seconds saved.
