The 6 Subsystems

① Text Parsing parser.js

Converts raw OPSL text into structured JavaScript objects. Uses simple string replacement — no regex.

How it works

The splitBy() function takes a string and a list of delimiters. It replaces each delimiter with |, then splits on |. This normalizes all separators into one common character.

// "VARIABLE test.x = "hello"" // Step 1: Replace ' = ' with '|' → 'VARIABLE test.x|"hello"' // Step 2: Replace '"' with '|' → 'VARIABLE test.x||hello|' // Step 3: Split on '|' → ['VARIABLE test.x', '', 'hello', ''] // Step 4: Map indices to fields → {type:'VARIABLE', library:'test', name:'x', value:'hello'}

Output

Each instruction becomes a plain JavaScript object with a type field and the relevant data:

InstructionObject Shape
IMPORT{type:'IMPORT', lang:'js', library:'test'}
VARIABLE{type:'VARIABLE', library:'test', name:'x', value:'hello'}
ASSIGN{type:'ASSIGN', library:'test', name:'result', libraryValue:'add', variableValue:'output'}
CALL{type:'CALL', library:'test', name:'sayHello'}

③ TCP Communication tcp-workers.js + tcp-server-v3/

All inter-process communication happens through TCP. A central server on port 3000 routes messages between library clients.

Key Concepts

  • Persistent Socket: One TCP connection opened once, reused for every instruction. ~0.06ms per instruction vs ~100ms if reconnecting.
  • Message Format: requestId:name:action:key[:value]\n — newline-delimited, colon-separated strings.
  • Request Tracking: Each request gets a unique ID. Responses are matched back to pending promises by ID.
  • Remote Calls: When library A calls B's function, the server opens a new TCP connection to B's local server, sends the call, reads the result, and returns it.

⑤ Data Management TCP Server + Client API

Variables are stored on the TCP server under each library's namespace. All values are strings — no type conversion.

Storage Model

// Server-side: clients = { test: { values: {myVar: "hello", count: "42"}, functions: ["sayHello"], port: 4001 }, add: { values: {number: "5", output: "5"}, functions: ["add"], port: 4010 }, ifJS: { values: {output: "LoopTest.loop", number: "5"}, functions: ["ifFunc"], port: 4011 } }

In-Function Variable Updates

When a function modifies a variable, it MUST call TCPClient.set() to update the server. Local-only changes are invisible to other libraries.

async function increment() { var current = await TCPClient.get('counter'); // Read from server var newVal = String(Number(current) + 1); // Modify locally await TCPClient.set('counter', newVal); // Update server }

⑥ File References Source files

Each library file is a standalone Node.js script that uses the TCPClient API. The runtime finds libraries by searching two locations.

Search Order

  1. projectFolder/libraryName.js — project root (checked first)
  2. opsl-local/libraryName/libraryName.js — local library folder (fallback)

Process Model

At any given time during execution, these processes exist:

ProcessStarted ByPurpose
node runtimeTCP/main.jsYou (the user)Main runtime process — orchestrates everything
node tcp-server-v3/server/index.jsserver-manager.jsTCP server on port 3000 — routes all messages
node library1.jstcp-workers.jsLibrary 1 — has its own TCP client + local server
node library2.jstcp-workers.jsLibrary 2 — same pattern, different port

Each library process connects to the TCP server AND runs its own local TCP server for receiving function calls from other libraries.

Performance

MetricValue
Per-instruction latency~0.06ms (60 microseconds)
1000-iteration loop~0.7 seconds total
Startup overhead~500ms (server startup + port cleanup)
TCP round-trip~0.02ms (localhost)
Why so fast? The persistent socket eliminates reconnection overhead. Each instruction reuses the same TCP connection, avoiding the ~100ms cost of opening/closing connections per instruction.
